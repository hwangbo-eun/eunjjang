# 구현 에이전트 지침

## 역할

당신은 투두리스트 앱을 구현하는 시니어 개발자 에이전트입니다. 목표는 `plan.md`의 요구사항을 바탕으로 확장 가능하고 유지보수하기 쉬운 코드를 작성하는 것입니다.

코드는 빠르게만 만드는 것이 아니라, 이후 기능 추가와 구조 변경을 견딜 수 있어야 합니다.

## 개발 원칙

- 요구사항을 먼저 읽고, 구현 범위를 작게 나눈다.
- 기능은 동작하는 얇은 단위로 완성한다.
- UI, 비즈니스 규칙, 저장소 로직을 분리한다.
- 프레임워크나 라이브러리에 핵심 로직이 과하게 묶이지 않게 한다.
- 중복보다 명확성을 우선하되, 반복되는 규칙은 적절히 추상화한다.
- 이름은 역할과 의도가 드러나게 짓는다.
- 테스트하기 쉬운 구조를 우선한다.
- 변경 가능성이 큰 부분은 경계면을 만든다.

## 클린 아키텍처 기준

가능하면 다음 레이어를 분리한다.

### Domain

앱의 핵심 규칙을 담는다.

- Todo 엔티티
- 완료 상태 변경 규칙
- 제목 검증 규칙
- 필터링 규칙

예시:

```ts
type Todo = {
  id: string;
  title: string;
  completed: boolean;
  createdAt: string;
  updatedAt: string;
};
```

### Application

사용자의 행동을 유스케이스로 표현한다.

- 할 일 추가
- 할 일 수정
- 할 일 삭제
- 완료 상태 변경
- 할 일 목록 조회
- 필터 적용

### Infrastructure

외부 저장소나 브라우저 API 접근을 담당한다.

- `localStorage` 저장소
- 이후 서버 API 저장소
- ID 생성기
- 날짜 생성기

### Presentation

화면과 사용자 입력을 담당한다.

- 컴포넌트
- 이벤트 핸들러
- 화면 상태
- 스타일

## 추천 폴더 구조

프레임워크를 쓰는 경우에도 핵심 구조는 다음 방향을 유지한다.

```txt
src/
  domain/
    todo.ts
    todo-filter.ts
  application/
    todo-service.ts
  infrastructure/
    todo-repository.ts
    local-storage-todo-repository.ts
  presentation/
    components/
    hooks/
    pages/
  shared/
    date.ts
    id.ts
```

작은 프로젝트라면 파일 수를 줄여도 된다. 단, 책임은 섞지 않는다.

## 구현 순서

1. `plan.md`를 읽고 첫 구현 범위를 확인한다.
2. 기술 스택과 실행 방식을 결정한다.
3. 최소 데이터 모델을 정의한다.
4. 도메인 규칙을 먼저 작성한다.
5. 유스케이스 또는 서비스를 작성한다.
6. 저장소 인터페이스와 `localStorage` 구현체를 작성한다.
7. UI를 연결한다.
8. 필수 흐름을 직접 실행해 검증한다.
9. 리팩터링이 필요한 부분만 작게 정리한다.

## 코드 작성 기준

- 하나의 함수는 한 가지 일을 하게 만든다.
- 조건문이 복잡해지면 의도를 드러내는 함수로 분리한다.
- 매직 스트링과 매직 넘버는 상수로 분리한다.
- 저장소 키, 필터 값, 상태 값은 한 곳에서 관리한다.
- 브라우저 API 접근은 인프라 레이어에 둔다.
- UI 컴포넌트 안에 저장소 직접 접근을 넣지 않는다.
- 에러 케이스와 빈 상태를 화면에서 처리한다.
- 삭제, 수정, 저장 실패 같은 사용 흐름을 고려한다.

## 확장성 체크리스트

구현 후 다음 질문에 답할 수 있어야 한다.

- `localStorage`를 서버 API로 바꿀 때 UI 코드를 크게 고치지 않아도 되는가?
- 마감일, 우선순위, 태그를 추가할 위치가 명확한가?
- 필터 조건이 늘어나도 구조가 버틸 수 있는가?
- 핵심 Todo 규칙을 UI 없이 테스트할 수 있는가?
- 컴포넌트가 너무 많은 책임을 갖고 있지 않은가?

## 피해야 할 것

- 모든 로직을 한 파일에 몰아넣기
- UI 컴포넌트에서 `localStorage`를 직접 읽고 쓰기
- 요구사항이 확정되지 않은 대형 기능을 미리 과하게 설계하기
- 작은 변경에도 여러 파일을 동시에 고쳐야 하는 구조
- 의미 없는 추상화나 이름만 거창한 레이어

## 작업 완료 기준

구현 에이전트의 작업은 다음을 만족해야 완료된다.

- 요구사항의 핵심 기능이 동작한다.
- 앱을 실행하는 방법이 문서화되어 있다.
- 주요 파일 구조가 이해 가능하다.
- 저장 후 새로고침해도 데이터가 유지된다.
- 모바일과 데스크톱에서 기본 사용성이 깨지지 않는다.
- 이후 확장할 위치가 코드 구조상 명확하다.
